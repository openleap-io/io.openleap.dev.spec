## SS1. Children & Variability Structure

### 1.1 Group Logic

**Group type:** mixed

**Business rationale for this structure:**

Portfolio Dashboard, Program Management, Lifecycle & Gate Management, and Gate Review Workflow are mandatory within this composition because they form the minimum portfolio governance framework — organizations need a portfolio view, program grouping, lifecycle control, and gate reviews. Project Scoring & Prioritization and Multi-Project Roadmap are optional strategic planning enhancements.

### 1.2 Mandatory Children

The following children are mandatory — they MUST be included whenever this composition node is selected:

- **F-PS-006-01** (Portfolio Dashboard): Required because it is part of the minimum viable portfolio & program capability.
- **F-PS-006-02** (Program Management): Required because it is part of the minimum viable portfolio & program capability.
- **F-PS-006-03** (Lifecycle & Gate Management): Required because it is part of the minimum viable portfolio & program capability.
- **F-PS-006-04** (Gate Review Workflow): Required because it is part of the minimum viable portfolio & program capability.

### 1.3 Optional Children

The following children are optional — they MAY be included based on product configuration:

- **F-PS-006-05** (Project Scoring & Prioritization): Optional enhancement that adds value but is not required for basic portfolio & program functionality.
- **F-PS-006-06** (Multi-Project Roadmap): Optional enhancement that adds value but is not required for basic portfolio & program functionality.

## SS3. Rationale & Decisions

### Why This Composition Node Exists

The Portfolio & Program composition groups features that share the `ps-prt-svc` backend domain. These features collectively implement the portfolio & program capability area of the PS suite. Grouping them under a single composition node allows:

1. **Product selection:** A product can include or exclude the entire portfolio & program capability as a unit
2. **Mandatory minimum:** Within the composition, mandatory children ensure a viable minimum
3. **Optional extensions:** Optional children allow tailoring to organizational maturity

### Alternatives Considered

1. **Flat feature list (no composition):** Rejected because it would lose the ability to include/exclude the capability as a group and would make mandatory/optional relationships harder to express in UVL.
2. **Deeper nesting (sub-compositions):** Rejected because the current 6 children are manageable as a flat list under one composition. If the feature count grows beyond 12-15, re-evaluate.

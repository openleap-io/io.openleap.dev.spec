## SS1. Children & Variability Structure

### 1.1 Group Logic

**Group type:** mixed

**Business rationale for this structure:**

Time Booking Entry, Cost Entry, Time/Cost Approval Workflow, and Time & Cost Reports are mandatory because time recording is the primary input for cost tracking and project progress measurement — without these, the budget and EVM features in ps-bud-svc have no actual data. My Time (Personal Dashboard) is optional because it provides a convenience view for individual users but is not required for organizational time tracking.

### 1.2 Mandatory Children

The following children are mandatory — they MUST be included whenever this composition node is selected:

- **F-PS-004-01** (Time Booking Entry): Required because it is part of the minimum viable project time & costs capability.
- **F-PS-004-02** (Cost Entry): Required because it is part of the minimum viable project time & costs capability.
- **F-PS-004-03** (Time/Cost Approval Workflow): Required because it is part of the minimum viable project time & costs capability.
- **F-PS-004-04** (Time & Cost Reports): Required because it is part of the minimum viable project time & costs capability.

### 1.3 Optional Children

The following children are optional — they MAY be included based on product configuration:

- **F-PS-004-05** (My Time (Personal Dashboard)): Optional enhancement that adds value but is not required for basic project time & costs functionality.

## SS3. Rationale & Decisions

### Why This Composition Node Exists

The Project Time & Costs composition groups features that share the `ps-tim-svc` backend domain. These features collectively implement the project time & costs capability area of the PS suite. Grouping them under a single composition node allows:

1. **Product selection:** A product can include or exclude the entire project time & costs capability as a unit
2. **Mandatory minimum:** Within the composition, mandatory children ensure a viable minimum
3. **Optional extensions:** Optional children allow tailoring to organizational maturity

### Alternatives Considered

1. **Flat feature list (no composition):** Rejected because it would lose the ability to include/exclude the capability as a group and would make mandatory/optional relationships harder to express in UVL.
2. **Deeper nesting (sub-compositions):** Rejected because the current 5 children are manageable as a flat list under one composition. If the feature count grows beyond 12-15, re-evaluate.

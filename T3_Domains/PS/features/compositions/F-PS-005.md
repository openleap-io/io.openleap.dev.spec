## SS1. Children & Variability Structure

### 1.1 Group Logic

**Group type:** mixed

**Business rationale for this structure:**

Persona Management, Staffing Slot Management, and Team Planner View are mandatory within this composition because they represent the minimum staffing workflow — define what roles are needed, create demand entries, and visualize the team plan. Demand Aggregation Dashboard and Scenario Planning are optional advanced capabilities for portfolio-level resource planning.

### 1.2 Mandatory Children

The following children are mandatory — they MUST be included whenever this composition node is selected:

- **F-PS-005-01** (Persona Management): Required because it is part of the minimum viable project staffing capability.
- **F-PS-005-02** (Staffing Slot Management): Required because it is part of the minimum viable project staffing capability.
- **F-PS-005-03** (Team Planner View): Required because it is part of the minimum viable project staffing capability.

### 1.3 Optional Children

The following children are optional — they MAY be included based on product configuration:

- **F-PS-005-04** (Demand Aggregation Dashboard): Optional enhancement that adds value but is not required for basic project staffing functionality.
- **F-PS-005-05** (Scenario Planning): Optional enhancement that adds value but is not required for basic project staffing functionality.

## SS3. Rationale & Decisions

### Why This Composition Node Exists

The Project Staffing composition groups features that share the `ps-res-svc` backend domain. These features collectively implement the project staffing capability area of the PS suite. Grouping them under a single composition node allows:

1. **Product selection:** A product can include or exclude the entire project staffing capability as a unit
2. **Mandatory minimum:** Within the composition, mandatory children ensure a viable minimum
3. **Optional extensions:** Optional children allow tailoring to organizational maturity

### Alternatives Considered

1. **Flat feature list (no composition):** Rejected because it would lose the ability to include/exclude the capability as a group and would make mandatory/optional relationships harder to express in UVL.
2. **Deeper nesting (sub-compositions):** Rejected because the current 5 children are manageable as a flat list under one composition. If the feature count grows beyond 12-15, re-evaluate.

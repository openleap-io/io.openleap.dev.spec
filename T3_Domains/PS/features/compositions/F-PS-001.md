## SS1. Children & Variability Structure

### 1.1 Group Logic

**Group type:** mixed

**Business rationale for this structure:**

The first seven children (Project List, Create/Edit, WBS Editor, Gantt, Work Package Detail, Dependency Management, Milestone Tracking) are mandatory because they form the minimum viable project management capability — without any one of them, the core planning workflow is incomplete. The remaining three (Baseline Management, Project Templates, Work Package Relations View) are optional enhancements that add value but are not required for basic project planning.

### 1.2 Mandatory Children

The following children are mandatory — they MUST be included whenever this composition node is selected:

- **F-PS-001-01** (Project List & Dashboard): Required because it is part of the minimum viable project management capability.

- **F-PS-001-03** (WBS Editor): Required because it is part of the minimum viable project management capability.

## SS3. Rationale & Decisions

### Why This Composition Node Exists

The Project Management composition groups features that share the `ps-prj-svc` backend domain. These features collectively implement the project management capability area of the PS suite. Grouping them under a single composition node allows:

1. **Product selection:** A product can include or exclude the entire project management capability as a unit
2. **Mandatory minimum:** Within the composition, mandatory children ensure a viable minimum
3. **Optional extensions:** Optional children allow tailoring to organizational maturity

### Alternatives Considered

1. **Flat feature list (no composition):** Rejected because it would lose the ability to include/exclude the capability as a group and would make mandatory/optional relationships harder to express in UVL.
2. **Deeper nesting (sub-compositions):** Rejected because the current 10 children are manageable as a flat list under one composition. If the feature count grows beyond 12-15, re-evaluate.

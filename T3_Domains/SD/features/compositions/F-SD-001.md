## SS0. Identity

### 0.1 Purpose
This composition node groups all capabilities for managing the full lifecycle of a sales order, from initial creation through status tracking and amendment. A sales representative uses these features to capture customer demand, monitor fulfillment progress, and adjust orders within policy constraints.

## SS1. Children & Variability Structure

### 1.1 Group Logic
**Group type:** mixed

**Business rationale:** Create Sales Order and Order Status Tracking are mandatory because any deployment needs the minimum ability to capture demand and monitor fulfilment. Order Amendment is optional because organizations with strict change-control policies may disable it to prevent post-confirmation modifications, relying instead on cancellation and re-order.

### 1.2 Sub-Groups

| Sub-group | Children | Group type | Rationale |
|---|---|---|---|
| Core order operations | `001-01`, `001-02` | mandatory | Minimum viable sales order processing |
| Change management | `001-03` | optional | Only enabled where order amendment policy permits in-flight changes |

### 1.3 Rationale for Tree Position
Order Management is the entry point for all downstream SD operations. Shipping & Delivery (F-SD-002) and Billing & Invoicing (F-SD-003) depend on confirmed sales orders from this composition. It exists as a distinct composition because it maps directly to the sd.ord bounded context and its aggregate root (SalesOrder).

## SS2. Constraints

### 2.1 Intra-Node Constraints
| If | Then | Rationale |
|---|---|---|
| `F-SD-001-03` is included | `F-SD-001-02` must be included | Order Amendment requires a visible order status view so users can identify which orders are eligible for amendment |

### 2.2 Cross-Node Constraints
| If | Then | Rationale |
|---|---|---|
| `F-SD-002` (Shipping & Delivery) is included | `F-SD-001` must be included | Deliveries are created from confirmed sales orders; no order means no delivery |
| `F-SD-003` (Billing & Invoicing) is included | `F-SD-001` must be included | Billing runs trigger from delivered sales orders; the order reference is mandatory on every invoice |

## SS3. Quality & Testing Guidance
All leaf features MUST be independently testable. Integration tests MUST verify that a sales order created in F-SD-001-01 appears immediately in F-SD-001-02 with status DRAFT. Amendment tests in F-SD-001-03 MUST verify that the `sd.ord.sales-order.amended` event is published after a successful PATCH.

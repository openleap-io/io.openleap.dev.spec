## SS0. Identity

### 0.1 Purpose
This composition groups all capabilities for defining overhead keys, executing overhead allocation cycles, and simulating allocations before commitment. A controlling specialist uses these features to configure cost distribution rules and run periodic overhead postings from cost centers to cost objects (orders, projects, products).

## SS1. Children & Variability Structure

### 1.1 Group Logic
**Group type:** mixed

**Business rationale:** Define Overhead Keys and Run Allocation Cycle are mandatory because the overhead module is meaningless without both configuration and execution. Allocation Simulation is optional because it is a planning aid — smaller deployments may run allocations without a pre-check simulation step.

### 1.2 Sub-Groups

| Sub-group | Children | Group type | Rationale |
|---|---|---|---|
| Core allocation | `002-01`, `002-02` | mandatory | Overhead management requires configuration and execution |
| Planning aid | `002-03` | optional | Simulation adds value only when business requires pre-flight checks |

### 1.3 Rationale for Tree Position
Overhead & Allocation Management sits downstream of Cost Center Management (F-CO-001). It consumes cost center structures and cost element definitions to apply overhead rates. It feeds co.pc (product costing) and co.pa (profitability analysis) with fully loaded costs.

## SS3. Quality & Testing Guidance
All leaf features MUST be independently testable. Integration tests MUST verify that executing an allocation cycle in F-CO-002-02 produces debit/credit postings visible in cost center balances (co-cca-svc). Simulation results MUST match actual allocation results when run with identical parameters.
